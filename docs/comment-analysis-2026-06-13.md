## 总体判断

评论区的主信号非常集中：用户不是单纯觉得“有趣”，而是大量表达“我也要学”“求教程”“求救”“蹲教程”“怎么做”。这说明上一条视频的核心转化点不是娱乐或炫技，而是强工具需求。

下一条视频应该直接兑现评论区期待：

> 上条评论区都在求教程，今天把这个文献晨报 Skill 的开源用法讲清楚。

## 主要评论类型

### 1. 求教程 / 求救 / 蹲教程

这是最强信号。截图中反复出现：

- `求教程`
- `求求教程`
- `跪求教程`
- `师兄求教程`
- `蹲教程`
- `怎么做呀，求帮助`
- `这个必须克隆学会啊`
- `后期求更新`

含义：

- 用户已经把你定位成“能教我搭科研自动化工作流的人”。
- 下一条如果只是展示开源仓库，不够；必须给清晰的第一步操作路径。
- 标题里应该出现“教程来了”或“评论区求的教程”。

### 2. 身份认同：师兄 / 同门 / 科研人

评论里多次出现类似：

- `师兄`
- `同门`
- `同门来了`
- `同门救救`

含义：

- 你的人设不是纯技术博主，而是“博士/科研同门带路”。
- 这种亲近感可以降低付费转化阻力。
- 后续文案不宜太商业化，应该继续保持“我把自己用的工作流整理出来给大家抄作业”的姿态。

### 3. 具体需求：怎么做 / 链接 / 能不能学

截图里可见用户在问：

- `可以分享一下链接吗`
- `怎么做`
- `能不能教`
- `我能学会吗`

含义：

- 下一条视频必须解决“入口在哪里”和“不会代码怎么办”。
- 评论区置顶要同时给两个路径：
  - 会配置：GitHub 开源地址。
  - 不会配置：评论/私信 `内测`，进入邀请码名单。

### 4. 情绪型支持：太牛 / 厉害 / 震惊

这类评论用于证明内容有吸引力，但商业价值低于“求教程”类评论。

含义：

- 可作为开头社会证明：“上一条很多同学在评论区问怎么做。”
- 不应作为下一条主要选题方向。

### 5. 表情包和玩梗

截图中有若干猫图、狗图、表情包回复。

含义：

- 评论区氛围轻松，说明“科研工具”内容没有显得过度严肃。
- 下一条教程仍然可以保留轻松表达，但教程信息密度要高。

## 对下一条视频的标题修正

优先标题：

> 评论区都在求教程：上条 9.1 万播放的文献晨报工作流，我开源了

次优标题：

> 上条视频大家求的文献晨报教程来了，任何专业都能改成自己的版本

不推荐标题：

> literature-morning-report 开源项目介绍

原因：太像技术项目说明，不能承接评论区的“求教程”情绪。

## 下一条视频必须回答的 4 个问题

1. 这个东西到底能不能给我的专业用？
2. 我应该填哪些信息？
3. 不会代码能不能用？
4. GitHub 地址在哪里，内测怎么申请？

## 评论区承接话术

置顶评论建议：

> 评论区求的教程来了。  
> 会配置的同学：GitHub 搜索 `literature-morning-report`，按 README 改 `research_profile.yml`。  
> 不会配置 API key / 邮箱 / 定时任务的同学：评论 `内测`，我先整理第一批托管版邀请码。  
> 想看数据库适配，评论 `数据库`；想看 EndNote/Zotero 导出，评论 `导出`。

私信自动回复建议：

关键词：`教程`

> 教程路径是：先看 GitHub README，再复制 `research_profile.template.yml` 改成自己的研究方向。下一条视频我会演示最短流程：改画像 → dry-run → 生成文献晨报。

关键词：`内测`

> 内测版适合不想配置 API key、SMTP、定时任务的同学。你可以发我：专业 + 研究方向 + 常用邮箱 + 是否需要 EndNote/Zotero 导出。我先整理邀请码名单。

关键词：`数据库`

> 开源版默认支持 PubMed、Europe PMC、OpenAlex、Crossref、Semantic Scholar 等；Web of Science、Scopus、Embase 需要你或机构有 API 权限，不能承诺无授权访问。

## 运营判断

这条视频已经验证了三个需求：

1. 科研人愿意收藏“文献晨报自动化”。
2. 用户愿意把你当作“科研自动化教程”的持续账号关注。
3. 评论区已经自然产生托管版入口：不会配置的人会需要邀请码/代配置/会员服务。

下一条的目标不是单纯冲播放，而是把“求教程”的人导入 GitHub 和内测名单。
